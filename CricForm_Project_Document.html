01
Project Overview & Vision

1.1 What is CricForm?

CricForm is a full-stack cricket analytics platform that computes a proprietary Form Score (0–100) for every active international cricketer. Unlike existing platforms such as Cricbuzz or ESPNcricinfo that display raw statistics, CricForm interprets those statistics using a weighted algorithm that accounts for recency, format, bowling/batting opposition quality, match context, and consistency.

1.2 The Core Differentiator

No cricket website publishes a single "form score" number with breakdown. They show data — you interpret it yourself. We do the interpretation systematically, consistently, for every player. The algorithm is the product.
ProblemSolution
Raw stats only, user interprets manuallyCricForm Form Score: one number, 0–100, fully explained
Same runs against AFG vs AUS treated equallyBowling Attack Multiplier adjusts for opposition quality
Inconsistent players look same as consistent onesConsistency Bonus based on standard deviation
World Cup final treated same as dead rubberMatch Context Multiplier (1.3x for finals)

02
Algorithm: Batter Form Score

2.1 Philosophy

The Batter Form Score judges a player purely on how they performed against the quality of opposition they faced. External conditions (pitch, venue, weather) are deliberately excluded. We judge the player, not the conditions.

Key insight: match-sequence weighting instead of time-based weighting. A player rested for 2 months should not be penalized. We only care about their last N matches played, regardless of when those matches occurred.

2.2 Complete Formula

BatterFormScore = NORMALIZE_0_100( SUM over last N matches of: RunsScored x SequenceWeight(matchRank) x FormatMultiplier(format) x BowlingAttackMultiplier(matchId) x MatchContextMultiplier(matchType) + ConsistencyBonus(lastNScores) )

2.3 Sequence Weight

Most recent match = rank 1. Weight decreases linearly. Never goes to zero.

SequenceWeight(rank) = 1.0 - ((rank - 1) * 0.1) rank 1 (most recent) = 1.0 rank 2 = 0.9 rank 3 = 0.8 rank 4 = 0.7 rank 5 = 0.6 rank 6 = 0.5 rank 7 = 0.4 rank 8 = 0.3 rank 9 = 0.2 rank 10 (oldest) = 0.1 Default N = 10. UI allows N = 5 or N = 15.

2.4 Format Multiplier

FormatMultiplierReason
T20I1.3Scoring quickly in T20 is hardest — higher reward
ODI1.1Run rate pressure adds difficulty
Test1.0Base multiplier

2.5 Bowling Attack Multiplier (Core Innovation)

Not all runs are equal. Runs scored against a peak bowling attack are worth more than runs scored against a weak attack. The Bowling Attack Score uses bowlers' own form at the time of that match (their last 5 matches before this game).

BowlingAttackScore(matchId) = AVERAGE of BowlerFormScore(bowlerId, asOfMatchDate) for each bowler who bowled in matchId BowlingAttackMultiplier: Score < 30 → 0.8 (weak attack) Score 30-45 → 0.9 Score 45-60 → 1.0 (average attack) Score 60-75 → 1.1 Score 75-85 → 1.2 Score > 85 → 1.3 (elite attack)

2.6 Match Context Multiplier

Match TypeMultiplier
World Cup Final / ICC Final1.3
Knockout / Semi-Final1.2
Regular Group Stage1.0
Dead Rubber (series decided)0.8

2.7 Consistency Bonus

A player scoring 38, 41, 39, 42, 40 is more valuable than one scoring 90, 4, 80, 2, 3. Measured using standard deviation of last N scores.

StdDev = standard deviation of RunsScored across last N matches ConsistencyBonus: StdDev < 10 → +10 points (extremely consistent) StdDev 10-20 → +7 points StdDev 20-30 → +4 points StdDev 30-40 → +2 points StdDev > 40 → +0 points

2.8 Normalization & Caps

NormalizedScore = ((RawScore - MinRawScore) / (MaxRawScore - MinRawScore)) * 100 Hard cap: score cannot exceed 100 or go below 0 Multiplier stack cap: total combined multipliers capped at 2.0x per match

2.9 Recursive Calculation Order

  • Step 1: Calculate ALL bowler scores using raw bowling stats only (no batter dependency). This is Iteration 0.
  • Step 2: Using Iteration 0 bowler scores, calculate ALL batter scores. This is Iteration 1.
  • Step 3: Using Iteration 1 batter scores, recalculate ALL bowler scores. This is Iteration 2.
  • Step 4: STOP. Do NOT iterate further. 2 iterations is sufficient. Prevents infinite loops.
  • Step 5: Store final scores to MySQL and cache in Redis.
03
Algorithm: Bowler Form Score

3.1 Complete Formula

BowlerFormScore = NORMALIZE_0_100( SUM over last N matches of: MatchBowlingScore(matchId) x SequenceWeight(matchRank) x FormatMultiplier(format) x BattingLineupMultiplier(matchId) x MatchContextMultiplier(matchType) + ConsistencyBonus(lastNMatchBowlingScores) )

3.2 Match Bowling Score

A bowler's single match performance = wickets + economy + overs + maidens. Economy is phase-adjusted — death economy of 9.0 is acceptable, powerplay economy of 9.0 is terrible.

MatchBowlingScore = WicketsComponent + EconomyComponent + OversComponent + MaidenBonus WicketsComponent = Wickets x WicketWeight Bowled = 15 pts (hardest to achieve) LBW = 14 pts Caught = 12 pts Stumped = 10 pts (batter error involved) Run Out = 0 pts (not bowler's credit) EconomyComponent (by match phase): Powerplay (overs 1-6): Economy < 5.0 → 30 pts Economy 5-7 → 20 pts Economy 7-9 → 10 pts Economy > 9 → 2 pts Middle overs: Economy < 6.0 → 25 pts Economy 6-8 → 15 pts Economy 8-10 → 8 pts Economy > 10 → 2 pts Death overs (16-20 T20 / 41-50 ODI): Economy < 8.0 → 30 pts (exceptional) Economy 8-10 → 18 pts Economy 10-12 → 10 pts Economy > 12 → 3 pts Test cricket: Economy < 3.0 → 30 pts Economy 3-4 → 20 pts Economy 4-5 → 10 pts Economy > 5 → 2 pts OversComponent: Bowled full quota → +10 pts Bowled 75%+ quota → +5 pts Bowled < 50% quota → +0 pts MaidenBonus = Maidens x 5 pts each

3.3 Batting Lineup Multiplier

Symmetric to BowlingAttackMultiplier. A bowler who dismisses in-form top batsmen gets higher credit.

BattingLineupScore(matchId) = AVERAGE of BatterFormScore(batterId, asOfMatchDate) for each batter who batted in that innings Same multiplier scale: 0.8 (weak lineup) to 1.3 (elite lineup)

3.4 All-Rounder Handling

AllRounderFormScore = (BatterFormScore x 0.5) + (BowlerFormScore x 0.5) Stored as: player.allRounderFormScore UI shows: both individual scores + combined score
04
Edge Cases & Rules

4.1 Batter Edge Cases

Minimum Matches Threshold

  • Minimum 5 matches required to generate a valid Form Score
  • If fewer than 5: store formScore = null, display "Insufficient Data" on UI
  • Player still appears in search but not on leaderboard

Not Out Innings

  • Not out innings (e.g., 45*) get a +5 bonus added to that match's contribution
  • Player preserved their wicket — has team value
  • Flag in DB: innings_not_out = true/false

Duck (Score = 0)

  • Run out for 0: mild penalty — multiply that match score by 0.8
  • Bowled/LBW/Caught for 0: standard zero, no extra penalty
  • Consecutive ducks: std deviation increases, naturally lowers consistency bonus

Did Not Bat

  • Match skipped entirely — not counted in last N matches window
  • Do not record this match in batting_innings table

Retired Hurt While Batting

  • Player was attempting to bat and got injured — not their fault
  • Prorate their score: ActualScore / (BallsFaced / AverageInningsLength)
  • Set innings_retired_hurt = true in DB, apply +3 bonus for intent

Retired Hurt While Fielding

  • Their batting innings was already complete
  • Count their batting innings fully — no modification

4.2 Bowler Edge Cases

Minimum Overs Threshold

  • Bowled fewer than 1 complete over: skip that match entirely
  • Emergency over is not representative bowling performance

Did Not Bowl

  • Skip that match entirely for bowling score
  • Do not record in bowling_innings table

Injured While Bowling Mid-Spell

  • Count only completed overs (round down)
  • Do not penalize for not completing quota
  • Set innings_injury_withdrawal = true in DB

Injured Before Bowling

  • Skip that match entirely for bowling score

4.3 Shared Rules

Format Separation

  • T20I, ODI, Test Form Scores are always calculated SEPARATELY
  • Never mix formats in a single score calculation
  • Leaderboard has three separate tabs: T20I / ODI / Test

Inactive Player

  • No matches in last 180 days: display "Out of Form Window"
  • Still searchable, profile shows last known score with date

Multiplier Stack Cap

  • Maximum combined multiplier per match = 2.0x
  • Prevents World Cup final + peak bowling attack from inflating unrealistically

5.5 Data Flow

1. Scheduler triggers at midnight IST 2. Fetch list of all recent international series 3. For each series, fetch all matchIds 4. For each matchId, fetch full scorecard 5. Parse scorecard: extract batting + bowling innings 6. For each batsman: upsert batting_innings table 7. For each bowler: upsert bowling_innings table 8. After all data ingested: trigger FormScoreCalculationService 9. Store calculated scores in player_form_scores table 10. Invalidate Redis cache for all updated players 11. Re-populate Redis with fresh scores

Redis Caching Strategy

8.1 What Gets Cached

DataReason
Player Form ScorePer player, per format — most frequently read
LeaderboardTop 10 per format — read on every page load
Player ProfileFull profile including last 10 match breakdown
Search IndexAll player names for fast autocomplete
Bowling Attack ScoresPer match — needed during algorithm recalculation

8.2 Redis Key Naming Convention

Player Form Score: KEY: cricform:player:{playerId}:form:{format} e.g: cricform:player:1413:form:ODI TTL: 86400 seconds (24 hours) Leaderboard: KEY: cricform:leaderboard:{type}:{format} e.g: cricform:leaderboard:batters:T20I TTL: 86400 seconds (24 hours) Player Profile: KEY: cricform:player:{playerId}:profile TTL: 86400 seconds (24 hours) Search Index: KEY: cricform:search:index TTL: 604800 seconds (7 days) Bowling Attack Score per Match: KEY: cricform:match:{matchId}:bowling_attack TTL: 604800 seconds (7 days) Last Updated Timestamp: KEY: cricform:last_refresh TTL: no expiry (manually updated by scheduler)

8.3 RedisConfig.java

@Configuration public class RedisConfig { @Bean public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) { RedisTemplate<String, Object> template = new RedisTemplate<>(); template.setConnectionFactory(factory); template.setKeySerializer(new StringRedisSerializer()); template.setValueSerializer(new GenericJackson2JsonRedisSerializer()); return template; } }

8.4 Cache Invalidation Strategy

  • After each nightly scheduler run: delete all cricform:player:* and cricform:leaderboard:* keys
  • After deletion: immediately repopulate all keys with fresh calculated scores
  • If API call fails during ingestion: do NOT invalidate cache — serve stale data
  • Stale data indicator: compare cricform:last_refresh timestamp with current time on every API response

8.5 Cache Miss Handling

On cache miss for player form score: 1. Query player_form_scores table in MySQL 2. If found: populate Redis cache, return data 3. If not found: trigger async recalculation for this player 4. Return 202 Accepted: "Score being calculated, retry in 30 seconds" On cache miss for leaderboard: 1. Query top 10 from player_form_scores ordered by form_score DESC 2. Populate Redis cache 3. Return data immediately

Scheduled Jobs

9.1 DailyRefreshScheduler.java

@Component public class DailyRefreshScheduler { @Scheduled(cron = "${cricform.scheduler.cron}") // midnight daily public void runDailyRefresh() { Step 1: dataIngestionService.runFullIngestionCycle() Step 2: formScoreService.recalculateAllScores() Step 3: cacheService.invalidateAllFormScores() Step 4: cacheService.repopulateAllFormScores() Step 5: cacheService.updateLastRefreshTimestamp() } }

9.2 Full Ingestion Cycle

DataIngestionService.runFullIngestionCycle(): 1. Call CricketApiService.fetchRecentInternationalSeries() - Fetch last 90 days of series - Extract all matchIds from all series 2. For each matchId: a. Check if match already exists in matches table b. If exists and ingested_at < 7 days ago: SKIP c. If new or recent: fetch full scorecard d. Parse batting innings → upsert batting_innings table e. Parse bowling innings → upsert bowling_innings table f. Upsert match record in matches table g. Upsert player records in players table 3. Rate limiting: add 500ms delay between each API call 4. On API error: log error, continue with next matchId

9.3 Score Recalculation Cycle

FormScoreService.recalculateAllScores(): ITERATION 0 — Bootstrap bowler scores: For each active bowler, for each format: Calculate raw bowler form score (wickets + economy only) No batting lineup multiplier in iteration 0 Store in memory: Map<Long, Map<String, Double>> ITERATION 1 — Calculate batter scores: For each active batter, for each format: Fetch last N batting_innings records For each innings: get BowlingAttackScore using iteration 0 bowler scores Apply full formula with all multipliers Store in memory map ITERATION 2 — Recalculate bowler scores: For each active bowler, for each format: Fetch last N bowling_innings records For each innings: get BattingLineupScore using iteration 1 batter scores Apply full formula Store as FINAL bowler score NORMALIZATION: Normalize all batter scores to 0-100 Normalize all bowler scores to 0-100 PERSIST: Upsert all scores into player_form_scores table
